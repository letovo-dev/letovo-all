# Chat API — инструкция для фронтенд-агента

Документ описывает все HTTP-эндпоинты модуля чатов: формат запросов, ответов, коды ошибок и поведенческие детали, которые важны при разработке UI.

---

## Аутентификация

Все эндпоинты (кроме явно помеченных) требуют заголовок:

```
Bearer: <token>
```

Токен выдаётся при `POST /auth/login`. Если заголовок отсутствует или токен невалиден — `401 Unauthorized`.

---

## 1. `GET /chats/`

Список пользователей, с которыми текущий пользователь может вести переписку.

**Правило включения в список:**
- есть запись в `chat_override` с `override_type = 'allow'` для пары → включается;
- нет записи, но у пользователя `chattable = true` → включается;
- есть запись с `override_type = 'block'` → **не включается** (даже если `chattable = true`).

**Ответ `200`:**

```json
{
  "result": [
    {
      "username": "alice",
      "display_name": "Alice",
      "avatar_pic": "/media/avatars/alice.png",
      "last_message": "Привет!",
      "last_message_time": "2026-04-27T14:00:00"
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `username` | string | Логин собеседника |
| `display_name` | string \| null | Отображаемое имя |
| `avatar_pic` | string \| null | Путь к аватару |
| `last_message` | string \| null | Текст последнего **не удалённого** сообщения |
| `last_message_time` | string \| null | ISO-timestamp последнего сообщения |

Сортировка: по `last_message_time DESC NULLS LAST`, затем по `username` алфавитно.

Мягко удалённые сообщения (`deleted_at IS NOT NULL`) **не учитываются** при выборке `last_message`.

## 2. `GET /chat/:username`

История переписки с конкретным пользователем. Поддерживает пагинацию.

**Query-параметры:**

| Параметр | По умолчанию | Ограничения | Описание |
|----------|-------------|-------------|----------|
| `limit` | `50` | 1–200 | Количество сообщений на страницу |
| `offset` | `0` | ≥ 0 | Смещение |

Пример: `/chat/alice?limit=20&offset=40`

**Ответ `200`:**

```json
{
  "result": [
    {
      "message_id": 42,
      "sender": "scv",
      "receiver": "alice",
      "message_text": "Привет!",
      "sent_at": "2026-04-27T14:00:00",
      "attachments": "https://example.com/file.png,https://example.com/doc.pdf"
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `message_id` | number | ID сообщения |
| `sender` | string | Логин отправителя |
| `receiver` | string | Логин получателя |
| `message_text` | string | Текст сообщения |
| `sent_at` | string | ISO-timestamp |
| `attachments` | string | Ссылки через запятую; пустая строка если нет вложений |

Сортировка: `sent_at DESC` (новые сначала). Учитывайте это при рендере: нужно реверсировать массив для отображения в хронологическом порядке.

Мягко удалённые сообщения (`deleted_at IS NOT NULL`) **не включаются** в ответ.

## 3. `POST /new_message`

Отправить сообщение.

**Тело запроса (JSON):**

```json
{
  "receiver": "alice",
  "text": "Привет!",
  "attachments": ["https://example.com/file.png"]
}
```

| Поле | Обязательно | Описание |
|------|-------------|----------|
| `receiver` | да | Логин получателя |
| `text` | да | Текст сообщения |
| `attachments` | нет | Массив URL-строк |

**Проверка разрешений перед отправкой:**
- `chat_override = 'block'` для пары → `403`
- нет override, но `receiver.chattable = false` → `403`
- `chat_override = 'allow'` → разрешено (даже если `chattable = false`)

**Ответ `200`:**

```json
{
  "message_id": 42,
  "sender": "scv",
  "receiver": "alice",
  "status": "sent"
}
```

**Коды ошибок:**

| Код | Причина |
|-----|---------|
| `400` | Отсутствует `receiver` или `text` |
| `401` | Нет или невалидный токен |
| `403` | Отправка запрещена (override block или receiver не chattable) |

---

## 4. `DELETE /chat/message/:id`

Мягкое удаление сообщения (устанавливает `deleted_at = NOW()`).

Разрешено: **автор сообщения** или **администратор**.

**Ответ `200`:**

```json
{
  "message_id": 42,
  "status": "deleted"
}
```

**Коды ошибок:**

| Код | Причина |
|-----|---------|
| `401` | Нет или невалидный токен |
| `403` | Текущий пользователь не является автором и не admin |
| `404` | Сообщение не найдено или уже удалено |

После удаления сообщение перестаёт появляться в `GET /chat/:username` и не учитывается как `last_message` в `GET /chats/`.

## 5. `POST /chat/permission` _(только admin)_

Установить или обновить override-правило для пары пользователей.

**Тело запроса (JSON):**

```json
{
  "user_a": "scv",
  "user_b": "alice",
  "override_type": "block"
}
```

| Поле | Описание |
|------|----------|
| `user_a` | Один из пользователей пары |
| `user_b` | Второй пользователь |
| `override_type` | `"allow"` или `"block"` |

Порядок `user_a`/`user_b` не важен — сервер сам выстраивает каноническую пару (`user_a < user_b`).

Если правило уже существует — оно перезаписывается (upsert).

**Ответ `200`:** `{"status":"ok"}`

**Коды ошибок:**

| Код | Причина |
|-----|---------|
| `400` | Неверный JSON, неизвестный `override_type`, или `user_a == user_b` |
| `401` | Нет или невалидный токен |
| `403` | Текущий пользователь не admin |
| `404` | Один из пользователей не существует |

---

## 6. `DELETE /chat/permission` _(только admin)_

Удалить override-правило для пары. После удаления логика возвращается к `chattable`.

**Тело запроса (JSON):**

```json
{
  "user_a": "scv",
  "user_b": "alice"
}
```

**Ответ `200`:** `{"status":"ok"}`

**Коды ошибок:**

| Код | Причина |
|-----|---------|
| `400` | Неверный JSON или `user_a == user_b` |
| `401` | Нет или невалидный токен |
| `403` | Текущий пользователь не admin |
| `404` | Правила для этой пары не существует |

---

## Логика разрешений — шпаргалка

```
can_chat(A, B):
  если override(A,B) = 'allow'  → ✅ разрешено
  если override(A,B) = 'block'  → ❌ запрещено
  иначе                         → B.chattable
```

Правило симметрично: `override(A,B)` и `override(B,A)` — одна и та же запись.

## Типовые UI-сценарии

### Экран списка чатов (`/chats/`)
1. `GET /chats/` → показать список.
2. Опционально: поллинг или WebSocket (на бэке не реализован) для обновлений.
3. Если `last_message = null` — чат ещё пуст; показать заглушку.

### Экран переписки (`/chat/:username`)
1. `GET /chat/:username?limit=50&offset=0` — загрузить первую страницу.
2. Реверсировать массив перед рендером (ответ идёт `DESC`).
3. Подгрузка истории вверх: увеличивать `offset` на `limit`.
4. `attachments` — строка с запятой-разделителем; для рендера: `attachments.split(',').filter(Boolean)`.

### Отправка сообщения
1. `POST /new_message` с `receiver`, `text`, опционально `attachments`.
2. При `403` — показать пользователю, что переписка заблокирована.
3. При `200` — добавить новое сообщение в UI без перезагрузки списка (используйте `message_id` из ответа).

### Удаление сообщения
1. `DELETE /chat/message/:id`.
2. При `200` — убрать сообщение из UI локально.
3. При `403` — сообщение чужое, скрыть кнопку удаления.
4. При `404` — сообщение уже удалено, обновить UI.

---

## Заметки по сериализации

- Все ответы — `Content-Type: application/json; charset=utf-8`.
- Числовые поля (`message_id`, `limit`, `offset`) — целые числа.
- Timestamps (`sent_at`, `last_message_time`) приходят в формате PostgreSQL (`2026-04-27 14:00:00`), не ISO 8601 со `T`-разделителем — учитывайте при парсинге (`new Date(str.replace(' ', 'T'))`).
- Поле `attachments` в `GET /chat/:username` — **строка**, не массив. Разбивайте по запятой на клиенте.
